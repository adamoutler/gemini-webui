# GEMWEBUI: High-Fidelity Ephemeral "Fake Gemini" Testing Sessions

## 1. Original Request Context
> "I want fake gemini accessed via new screen with a query string. after providing a query string once, it should give a gemini session string to use which is ephemeral. i want friction on reusing the session, not closing the session. The overall goal here is to make Fake Gemini use all the same resources as real gemini. the idea i want, to summarize, is how to ensure all fake gemini responses are as close to a real session as possible because we keep having problems and variances when using fake gemini such as not using xterm, input differences, and fake gemini in prod. we also have problems with fake gemini testing working when prod does not. so we need a true fake gemini session and way to test."

## 2. Problem Statement & Goal
Currently, the test harness bypasses actual execution paths (like WebSocket streaming, `xterm.js` rendering, and PTY handling), creating a "mocking gap." Tests pass while production fails because Fake Gemini does not share the same resource pipeline as Real Gemini.

**The Goal:** Eliminate testing variances by executing Fake Gemini as a standard binary within the exact same pseudo-terminal (PTY) lifecycle, WebSocket event loop, and frontend UI (`xterm.js`) as production. Test sessions must be ephemeral, explicitly isolated, and introduce intentional UX "friction" against state-polluting reuse.

---

## 3. Implementation Details & Technical Scope

This epic requires coordinated changes across the Backend, Frontend (UX), and QA test suite.

### Sub-task A: Backend & Process Management
*   **Unified PTY Process:** Stop mocking internal Python functions or bypassing SocketIO. Modify `process_manager.py` to allow dynamic injection of the target executable path. For test sessions, execute `python3 src/fake_gemini.py <args>` within the standard `pty.fork()` wrapper.
*   **Ephemeral Session Generation:**
    *   Create a new backend route: `GET /fake_session_init?scenario=X` (or similar).
    *   Generate a highly restrictive, cryptographically random `session_id` (e.g., UUIDv4) upon access.
    *   Map this `session_id` in memory to `executable_override="src/fake_gemini.py"` with the provided query string arguments.
    *   Automatically redirect the user to the main terminal interface using this ephemeral ID (e.g., `302 Redirect -> /?session_id=<UUID>&mode=fake`).
*   **Backend Reconnection Friction:**
    *   Only allow a *single* WebSocket connection to bind to this specific ephemeral `session_id`.
    *   If the WebSocket drops or the PTY process exits, purge the `session_id`. Subsequent reconnection attempts at the transport layer must result in a strict rejection (403/404).

### Sub-task B: UX & Frontend (`app.js` & Templates)
*   **New Entrypoint (`/test-launcher`):** Create a dedicated, clean UI for launching tests. Present an input field for the "Query String" scenario parameters. On submit, hit the backend initialization endpoint and follow the redirect.
*   **Identical xterm.js Rendering:** The frontend must use the standard terminal UI and SocketIO pipeline. Parse the `?session=` and `?mode=fake` URL parameters to bypass the standard connection list and instantly bind to the test session.
*   **Visual Distinction:** Apply a specific CSS class (e.g., `.theme-fake-session`) to test tabs to render a persistent visual indicator (e.g., a striped background, colored border, or "TEST MODE" watermark).
*   **UX Friction Modal:** Intercept page reloads (restoring from `sessionStorage`) and Socket disconnects/reconnects for fake sessions. Instead of silently reconnecting, obscure the terminal with an overlay modal warning the user about test pollution. Provide two options:
    1.  `[ Start Fresh Test ]` (Recommended -> routes back to `/test-launcher`)
    2.  `[ Force Reconnect ]` (Allows the socket to restart).

### Sub-task C: The Synthetic Binary (`fake_gemini.py`)
*   **CLI Behavior Parity:** The fake script must act identically to a real CLI application.
*   **I/O Handling:** It must read from `sys.stdin` character-by-character, parse passed scenario arguments, write to `sys.stdout`, and proactively flush output buffers.
*   **Rich Formatting:** It must emit raw ANSI escape codes for colors and box-drawing to accurately test `xterm.js` parser fidelity.

### Sub-task D: E2E Test Suite (QA)
*   **Playwright Integration:** Update the test suite to use the new `/test-launcher` or initialization endpoint.
*   **True E2E Validation:** Tests must wait for the WebSocket, type directly into the real `xterm.js` `<textarea>`, and assert against the DOM's terminal rows. This guarantees that ANSI parsing and WebSocket race conditions are validated against production code.

---

## 4. Strict Acceptance Criteria & Definition of Done

1.  **Architecture Parity:** `fake_gemini.py` runs inside `pty.fork()` and utilizes the exact same SocketIO endpoints and `xterm.js` renderer as the production `gemini` executable.
2.  **Session Ephemerality:** Fake sessions are granted a unique, one-time ID. Backend aggressively rejects overlapping WebSocket connections to the same ephemeral ID.
3.  **UI Visual State:** A user or test running a fake session sees a clear, undeniable visual indicator (`.theme-fake-session` styling).
4.  **UX Friction:** Triggering a page reload or socket disconnect on a fake session successfully prompts the overlay warning modal before allowing reconnection.
5.  **Binary Fidelity:** `fake_gemini.py` successfully reads standard input and emits complex ANSI-formatted output to standard out.
6.  **Tests Passing:** All existing and newly introduced Playwright E2E tests pass reliably without relying on internal backend mocking (other than the PTY binary swap).

## 5. Automated Tests and Visual Evidence Requirements

*   **Evidence of UI Distinction:** Provide screenshots to `/tmp/` showing the `.theme-fake-session` visual styling and the newly added `/test-launcher` screen.
*   **Evidence of UX Friction:** Provide a screenshot to `/tmp/` of the warning modal overlay triggered after a disconnect/refresh.
*   **E2E Test Output:** The Playwright test logs must be cleanly captured to demonstrate successful full-stack interactions without bypassing WebSockets or xterm.js.
