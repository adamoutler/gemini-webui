Here is the architectural plan to align the Fake Gemini environment with the production Gemini execution path, eliminating testing variances while satisfying your requirements for ephemeral sessions.

### Architectural Strategy
The core issue is that Fake Gemini and Real Gemini likely diverge in how their standard streams (stdin/stdout) are handled or how they are spawned. To achieve zero variance, **Fake Gemini must be executed as a standard binary within the exact same pseudo-terminal (PTY) lifecycle and WebSocket event loop as production.**

### Implementation Plan

**1. Unified Process Management (Zero Variance Execution)**
*   **The Change:** Stop mocking internal Python functions or bypassing SocketIO for Fake Gemini tests.
*   **Implementation:** Modify `process_manager.py` to allow dynamic injection of the target executable path on a per-session basis. By default, it uses the system `gemini` binary (or SSH command). For testing, it will explicitly execute `python3 src/fake_gemini.py <args>` within the `pty.fork()` wrapper.
*   **Why:** This ensures xterm.js, ANSI character encoding, latency, and WebSocket chunking behave identically to production because the transport layer is exactly the same.

**2. Ephemeral Session Generation (The New Screen/Entrypoint)**
*   **The Change:** Create a mechanism to spawn these isolated fake sessions via a query string.
*   **Implementation:**
    *   Create a new backend route: `GET /fake_session_init?scenario=X&...`
    *   When hit, the backend generates a highly restrictive, cryptographically random `session_id` (e.g., UUIDv4).
    *   The `SessionManager` registers this `session_id` in memory, explicitly flagging it with `executable_override="src/fake_gemini.py"` and storing the requested query parameters as CLI arguments for the fake binary.
    *   The backend immediately redirects the user to the main terminal interface using this ephemeral ID: `302 Redirect -> /?session_id=<UUID>`.

**3. Enforcing Session "Friction" (Single-Use Lifecycle)**
*   **The Change:** Make it difficult to reuse the session, but easy to discard it.
*   **Implementation:**
    *   **Single Connection Binding:** The backend will only allow a *single* WebSocket connection to bind to this specific `session_id`.
    *   **Burn on Disconnect:** If the WebSocket drops, the user navigates away, or the PTY process exits, the backend immediately purges the `session_id` from the active sessions dictionary.
    *   **Reconnection Rejection:** Any subsequent attempt to load `/?session_id=<UUID>` or reconnect the SocketIO client will result in a hard HTTP 403/404, forcing the user (or test suite) to generate a brand new session via `/fake_session_init`.

**4. `fake_gemini.py` as a True CLI**
*   **The Change:** `fake_gemini.py` must behave exactly like a real CLI application.
*   **Implementation:** It must read from `sys.stdin` and write to `sys.stdout` using standard blocking/async CLI loops. It should parse the query string arguments (passed to it by the Process Manager) to determine which "scenario" or mock data to output. It should emit raw ANSI escape codes just like the real Gemini binary.

### Summary of Data Flow
1. User/Test navigates to `/fake_session_init?mock_response=hello`
2. Backend generates `session_id=1234` -> maps it to `python3 fake_gemini.py --mock_response hello`.
3. Backend redirects to `/?session_id=1234`.
4. Frontend loads standard `xterm.js`, connects WebSocket to `session_id=1234`.
5. Backend calls `pty.fork()`, executing the fake binary. Streams are piped seamlessly.
6. User closes tab -> backend deletes `1234`. Session cannot be restored.
